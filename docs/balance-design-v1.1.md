## 一、核心数值公式

### 1.1 过路费计算公式（完整版）

```
应付过路费 = floor(
  城池基础过路费
  × 升级倍率[level]
  × 区域加成系数
  × 城主进攻技能系数
  × 访客防御技能系数
)

其中：
  城池基础过路费 = 购买价 × 10%
  
  升级倍率：
    1级 = 1.0
    2级 = 1.5
    3级 = 2.0
    4级 = 3.0
    5级 = 4.0
  
  区域加成系数：
    荆州 = 1.2
    其他 = 1.0
  
  城主进攻技能系数：
    张飞 = 1.5
    其他 = 1.0
  
  访客防御技能系数：
    刘备 = 0.5
    夏侯惇 = 0.7
    甘宁劫营（特殊）：访客实付 = 应付 × 0.5，城主实收 = 应付 × 0.5
    无技能 = 1.0
  
  floor：向下取整到整数
```

**计算示例**：

| 场景 | 公式展开 | 结果 |
|------|----------|------|
| 洛阳（2000金）5级，无加成，无技能 | 200 × 4 × 1.0 × 1.0 × 1.0 | 800 |
| 洛阳5级 + 张飞城主 + 刘备访客 | 200 × 4 × 1.0 × 1.5 × 0.5 | 600 |
| 荆州城池（1000金）3级 + 张飞 + 夏侯惇 | 100 × 2 × 1.2 × 1.5 × 0.7 | floor(252) = 252 |
| 甘宁踩洛阳5级 | 应付800，甘宁付400，城主收400 | 各400 |

### 1.2 城池升级费用与累计投入

```
升级费用[level] = 购买价 × 升级费率[level]

升级费率：
  1→2级 = 0.5
  2→3级 = 1.0
  3→4级 = 1.5
  4→5级 = 2.0

累计投入[level] = 购买价 × 累计系数[level]

累计系数：
  1级 = 1.0（仅购买价）
  2级 = 1.5
  3级 = 2.5
  4级 = 4.0
  5级 = 6.0

出售价[level] = floor(累计投入[level] × 0.5)
```

| 城池 | 购买价 | 2级升级费 | 3级升级费 | 4级升级费 | 5级升级费 | 5级总投入 | 5级出售价 |
|------|--------|-----------|-----------|-----------|-----------|-----------|-----------|
| 洛阳 | 2000 | 1000 | 2000 | 3000 | 4000 | 12000 | 6000 |
| 成都 | 1800 | 900 | 1800 | 2700 | 3600 | 10800 | 5400 |
| 虎牢关 | 1600 | 800 | 1600 | 2400 | 3200 | 9600 | 4800 |
| 建业 | 1400 | 700 | 1400 | 2100 | 2800 | 8400 | 4200 |
| 邺城 | 1200 | 600 | 1200 | 1800 | 2400 | 7200 | 3600 |
| 汉中 | 1000 | 500 | 1000 | 1500 | 2000 | 6000 | 3000 |
| 晋阳 | 900 | 450 | 900 | 1350 | 1800 | 5400 | 2700 |
| 陈留 | 800 | 400 | 800 | 1200 | 1600 | 4800 | 2400 |
| 绵竹 | 700 | 350 | 700 | 1050 | 1400 | 4200 | 2100 |
| 濮阳 | 600 | 300 | 600 | 900 | 1200 | 3600 | 1800 |

### 1.3 回本分析（城池购买的盈亏平衡点）

```
回本所需被踩次数[level] = 
  (累计投入[level] - 购买价) / (过路费[level] - 过路费[1])

注：分子是相比1级的额外投入，分母是相比1级的额外过路费收益
```

**洛阳回本分析**：

| 目标等级 | 额外投入 | 额外过路费/次 | 回本次数 | 评价 |
|----------|----------|---------------|----------|------|
| 升到2级 | 1000 | 100（300-200） | 10次 | 中等 |
| 升到3级 | 3000 | 200（400-200） | 15次 | 偏慢 |
| 升到4级 | 6000 | 400（600-200） | 15次 | 相同 |
| 升到5级 | 10000 | 600（800-200） | 17次 | 偏慢 |

> **结论**：高等级升级的回本时间较长，需要城池处于高频踩踏路段才合算。起点附近（洛阳）是必经之地，升级最优先。

## 二、各人数局资金流动预测

### 2.1 2人局资金模型

**初始状态**：
- 各玩家：12000金
- 城池总价值（8折）：32800 × 0.8 = 26240金
- 可购城池总数：32座

**每圈平均收益/支出估算**（稳定期，约第5圈起）：
| 项目 | 估算值 | 说明 |
|------|--------|------|
| 起点奖励 | 2000 | 固定 |
| 锦囊卡平均收益 | +150 | 约 2 个锦囊格/圈，平均价值 300×0.5（正负抵消） |
| 天命卡平均收益 | +100 | 偏正面，约 2 个天命格/圈 |
| 过路费支出 | -800 | 对方约 8-10 座城池，平均 1 级 |
| 过路费收入 | +800 | 对称 |
| **净现金流** | **+2250/圈** | 不含城池购买/升级 |

**预计破产时间线**：
- 前 3 圈：两方都在购买城池，资金消耗约 8000-10000
- 4-8 圈：双方资金趋于稳定（10000-14000），过路费逐渐拉开差距
- 8-15 圈：差距明显，落后方开始出售城池
- 15-25 圈：单方破产

**2人局时长估算**：20-35 圈 × 约 90 秒/圈 ≈ 30-52 分钟 ✓（目标 30-45 分钟）

### 2.2 4人局资金模型（标准）

**初始状态**：
- 各玩家：15000金（总共 60000）
- 城池总价值：32800金（人均拥有 8 座）
- 买城花费约 8200/人（平均分配时）

**稳定期每圈估算**：
| 项目 | 估算值 |
|------|--------|
| 起点奖励 | 2000 |
| 过路费收入 | +600（8座×100×0.75，被踩概率调整） |
| 过路费支出 | -1500（踩到 3 座他人城池，平均 100×3人） |
| 锦囊/天命净收益 | +100 |
| **净现金流** | **+1200/圈** |

**武将技能对资金流的影响（4人局，30圈）**：

| 武将 | 30圈累计技能收益估算 |
|------|---------------------|
| 曹操（15%起点加成） | 300 × 10圈 = 3000 |
| 荀彧（+100/回合） | 100 × 30 × 4 = 12000 **（注意：全局回合数×4人）** → 荀彧约 100×30 = 3000 |
| 许褚（初始+2000） | 2000（一次性） |
| 刘备（50%过路费减免） | 节省约 750（15次×100×50%） |
| 张飞（+50%收取） | 额外收取约 500 |

### 2.3 8人局资金模型

**初始状态**：
- 各玩家：20000金（总 160000）
- 城池数量：32座，人均 4 座
- 城池竞争激烈，部分城池可能在1圈内被抢购

**特殊考虑**：
- 8人游戏中，每人城池少，过路费收入不稳定
- 锦囊卡效果 +50%，战略价值大幅提升
- 加速模式（20回合后）显著加快终局
- 推荐「首先达到40000金」作为目标（否则游戏可能拖长）

### 4.3 关键参数敏感性分析

以下参数的微调对游戏体验影响最大，需要在测试中重点观察：

**高敏感度参数**（小幅变动 = 大影响）：
1. 起点奖励（2000金）：每 ±200 金影响约 ±8% 的整体资金流速
2. 过路费倍率（10%）：每 ±1% 影响约 ±5% 的玩家资产分化速度
3. 荆州区域 +20%：这是唯一的区域差异，是否保留需测试（可能导致玩家绕路）

**低敏感度参数**（可安全调整）：
1. 武将技能额外收益（±100金级别的调整几乎无感）
2. 天命卡正负面比例（小幅调整不影响体验）
3. 城池升级费率（回本分析显示高级升级本就不划算）

## 五、数值调整决策流程

```
发现某武将胜率异常
    │
    ├─ 胜率 > 35%？
    │       ├─ YES → 降低技能收益（减少固定值，约 -10%~-20%）
    │       └─ 先观察是否偶然性（至少 200 局样本）
    │
    ├─ 胜率 < 15%？
    │       ├─ YES → 提高技能收益，或降低使用门槛
    │       └─ 检查是否技能本身存在 Bug（技能未正常触发）
    │
    └─ 15% ≤ 胜率 ≤ 35% → 保持观察，不调整
```

**调整原则**：
1. 每次只调整 1 个参数（隔离变量）
2. 调整幅度不超过 ±20%（防止过度修正）
3. 调整后需再运行 200+ 局模拟验证
4. 优先调整数值，而非技能机制（避免破坏策略深度）
